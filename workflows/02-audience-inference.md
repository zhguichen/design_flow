# Workflow 2：人群反推 (audience-inference)

读入 Workflow 1 的问卷，分析每道题在测什么，反推"会让不同人答出不同答案的关键影响变量"，划出 5-8 类人群原型 + 比例。结果变量不能作为 persona 背景设定，只设定影响它们的原因。

按需从根 `SKILL.md` 加载。依赖 WF1 的 `construct_measured` 字段——缺它则触发 Stop。

## When to use

- 问卷（`survey.json`）已就绪，需要确定"要生成哪些类型的合成受访者"
- 用户想理解"什么样的人会对这份问卷答出不同结果"
- WF1 完成后由 pipeline 自动推进到本步

不适用：还没有结构化问卷（`construct_measured` 缺失）——先回 WF1。

## 输入

- `runs/<时间戳>/survey.json`：必须含每题 `construct_measured`。

## 输出

写 `runs/<时间戳>/archetypes.json`，schema：

```json
{
  "research_question": "...",
  "target_population": "...",
  "note": "合成样本 / 仅供预调研",
  "archetypes": [
    {
      "archetype_id": "A1",
      "name": "赶作业型学生",
      "核心特征": "时间压力高，方法意识中低，关心效率",
      "proportion": 0.30,
      "变量设定": [
        {"类型": "情境", "变量": "时间压力", "取值": "高", "来源": ["Q2"]},
        {"类型": "心理", "变量": "AI信任", "取值": "中高", "来源": ["Q3"]}
      ],
      "预期回答倾向": "整体定性描述",
      "预期影响construct": {"使用意愿": "高", "付费意愿": "低", "信任": "中"}
    }
  ]
}
```

字段：

- `archetype_id`：全文件唯一，`A1..An`。
- `proportion`：0-1 之间，所有原型合计 = 1.0。
- `变量设定`：每条带 `类型`（五类之一：经验 / 情境 / 心理 / 资源 / 能力）、`变量`、`取值`、`来源`（追溯到 `question_id` 列表）。
- `预期回答倾向`：**整体定性**描述（不逐题给答案，保留 WF4 有限理性空间）。
- `预期影响construct`：标注这个原型对各结果构念的方向（高 / 中 / 低），键取自 WF1 的 construct 词表。

## 方法

### 第一步：分析每题在测什么

逐题读 `construct_measured`，回答：

- 这道题测的是态度 / 行为 / 需求 / 信任 / 付费意愿 / 满意度 / 体验，还是基础身份？
- 什么样的人会答得不一样？
- 这种差异来自身份、经验、资源、心理、价值观，还是情境？
- **哪些是结果变量**（满意度 / 使用意愿 / 推荐意愿 / 付费意愿）——这些**不能**用来定义原型背景，只能作为被影响的后果。

### 第二步：提取关键影响变量（五类框架）

从研究问题里长出来，不是属性全排列。五类：

| 类型 | 说明 | 示例 |
|---|---|---|
| 经验 | 是否做过、是否遇过这事 | 做过真实调研 / 遇过问卷没人填 |
| 情境 | 当下处境与压力 | 时间压力 / 毕业设计阶段 / 导师要求 |
| 心理 | 倾向与态度 | AI 信任 / 造假担心 / 风险偏好 |
| 资源 | 有什么可用 | 社交资源 / 预算 / 真实用户渠道 |
| 能力 | 掌握度与熟练度 | 研究方法掌握度 / AI 工具熟练度 |

每个变量追溯到至少一个 `question_id`（哪道题会让这个变量显现差异）。

### 第三步：变量 → 人群原型（5-8 类）

不要属性全排列。正确链路：关键变量 → 人群原型 →（后续）具体个体。

原型要满足两条：

1. **能解释回答差异**：不同原型对同一道题会答出不同答案。
2. **现实合理存在**：不是凑数的拼凑组合，是真实存在于目标人群里的类型。

原型数 5-8。少于 5 覆盖不足，多于 8 失去聚合意义。

### 第四步：设定比例

`proportion` 合计 = 1.0。比例依据目标人群的现实分布（如"赶作业型"在学生里占比高）。WF2 自动提议，用户可调。比例是分层，不是样本量——样本量 = 比例 × N，N 在 WF3 定。

## 红线（核心）

- **结果变量不作背景设定**：满意度 / 使用意愿 / 推荐意愿 / 付费意愿是问卷要测的**结果**，不能提前塞进 `变量设定` 当 persona 背景。只设定影响它们的原因（经验 / 情境 / 心理 / 资源 / 能力）。
- `预期影响construct` 是**预测后果方向**，不是 persona 的背景设定——它描述"这类人大概会往哪个方向答"，供 WF3 / WF4 参考，WF4 仍须依 persona 处境真实作答。
- 全 pipeline 红线继承：本步输出是合成人群画像，文件级 `note` 字段标注 合成样本 / 仅供预调研。

## 验收标准（可机械检查）

不满足 = 未完成：

- [ ] 文件是合法 JSON，含 `research_question` / `archetypes`。
- [ ] 原型数 5-8。
- [ ] 每个 `archetype_id` 唯一。
- [ ] 所有 `proportion` 合计 = 1.0（±0.001 容差）。
- [ ] 每个原型有 `变量设定`，每条含 `类型`（五类之一）/ `变量` / `取值` / `来源`（非空 `question_id` 列表）。
- [ ] `来源` 里的 `question_id` 都能在 `survey.json` 找到。
- [ ] `变量设定` 的 `类型` 不含"满意度 / 使用意愿 / 推荐意愿 / 付费意愿"等结果构念（结果变量红线）。
- [ ] 每个原型有 `预期回答倾向`（整体定性，非逐题答案）+ `预期影响construct`。
- [ ] 每个原型有 `核心特征`。

## Stop 条件

- `survey.json` 不存在或无 `construct_measured` → 不臆造，回 WF1 补。
- 反推不出 5 类以上有意义的变量 → 说明问卷覆盖不足，回 WF1 加题。
- 用户给的目标人群与研究问题矛盾 → 追问澄清。

## 下一步

人群原型就绪后进入 `workflows/03-persona-generation.md`：按 `proportion` × N 展开成具体模拟受访者，每个带五层信息结构，同类下个体必须有差异。

    {
