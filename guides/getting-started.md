# design-flow 小白使用指南

这份指南写给第一次使用 Claude Code Skill、不了解 persona、行为机制或任务摩擦的人。你不需要会写代码，也不需要自己创建 JSON 文件。准备好一个研究问题，然后让 Agent 按流程生成即可。

## 先理解它能做什么

`design-flow` 是问卷预调研工具。它可以帮你设计问卷，构造一批覆盖不同情境的合成受访者，让这些模拟受访者填写问卷，再整理出一份分析报告。

它适合在正式找真人调研前做这些事：

- 检查问题是否难懂、重复或带有引导性。
- 看不同使用情境可能产生哪些答案差异。
- 提前发现值得在访谈或真实问卷中验证的假设。
- 课程作业时间紧、暂时找不到合适受访者时，替代低质量的同学互填。

它不能证明真实人群就是这样想，也不能提供市场比例、置信区间或论文级结论。报告中的人数和百分比只描述本次合成场景。

## 使用前准备三句话

开始前尽量想清楚下面三件事。说得不完整也没关系，Agent 会在必要时追问。

第一句是“我想研究什么”。例如：

> 我想知道城市租房者购买模块化家具时，最担心哪些问题。

第二句是“谁来回答”。不要只写年龄或性别，最好包含真实经历。例如：

> 近半年购买过家具、目前住在出租房的人。

第三句是“结果准备拿来做什么”。例如：

> 用来调整课程设计的产品功能优先级，之后还会找真人访谈。

如果你想“发现自己完全不知道的问题”，先做访谈更合适。问卷擅长确认和比较已经知道的构念，不擅长发现未知问题。

## 安装后怎么开始

确保 `design_flow/SKILL.md`、`workflows/`、`references/` 和 `scripts/` 保持在同一个 Skill 目录中，并且你的 Agent 能发现这个 Skill。

进入准备保存项目产出的工作区，对 Agent 输入：

```text
/design-flow:run-pipeline 城市租房者对模块化家具的需求与购买决策调研
```

也可以直接使用自然语言：

```text
请用 design-flow 帮我做一次问卷预调研。
我想研究近半年买过家具的城市租房者，在选购模块化家具时最在意什么。
结果用于课程设计，后续会找真人验证。
```

如果你已经有问卷，把问卷内容一起交给 Agent，并说明“请先检查和修订，不要重新从零设计”。

## 运行中你需要确认什么

完整流程只会在三个真正需要你做决定的地方暂停。

### 第一次：确认问卷

Agent 会展示完整题目。重点检查题目是不是你真正想问的、目标人群能不能理解、选项有没有漏掉常见情况。

此时可以直接说：

```text
Q4 太像引导用户购买了，请改成中性表达。
```

确认后，Agent 才会分析可能出现哪些不同类型的回答者。

### 第二次：确认人群分析和模拟规模

Agent 会展示人群原型、行为机制、任务情境和建议生成的合成记录数。

你不需要判断理论名称是否“高级”，主要看这些场景是否符合研究对象，是否遗漏了重要情境，是否出现明显矛盾。例如：

```text
短租用户和长期租房用户确实不同，但还缺少与家人合住、需要共同决策的人。
```

`simulation_n` 只是生成预算。记录更多可以增加同一场景下的变化，但不会增加真实世界证据。没有特别要求时，使用 Agent 给出的基础覆盖数量即可。

### 第三次：确认 persona

Agent 会展示几条 persona，并保证每个人群原型至少有一个完整例子。检查这些人是否像目标用户，经历和限制是否具体，同类之间是否真的有差异。

不要要求 Agent 预先写“这个人一定愿意购买”或“这个人满意度一定很低”。这些是问卷要测的结果，提前写进 persona 会让模拟变成自证。

确认后，模拟填写和结果分析会连续完成。

## 最后会得到什么

所有产出保存在 `runs/<时间戳>/`。通常最先阅读 `report.md`，其他文件用于追溯过程。

| 文件 | 小白可以怎么理解 |
|---|---|
| `survey.json` | 最终问卷 |
| `archetypes.json` | 为了覆盖不同情境而划分的人群原型 |
| `behavior_mechanisms.json` | 为什么这些情境可能影响回答 |
| `task_frictions.json` | 哪些具体经历、资源和环境限制值得写进 persona |
| `hypotheses.json` | 模拟前封存的预测，防止事后改答案 |
| `respondents.jsonl` | 合成受访者 |
| `responses.jsonl` | 合成作答 |
| `stats.json` | 脚本计算的描述统计 |
| `report.md` | 给设计者阅读的最终分析报告 |

`.jsonl` 表示“一行一条记录”。通常不需要手动编辑；如果你想调整 persona，让 Agent 修改并重新运行对应阶段。

## 怎么读最终报告

先看“这批人整体怎么想”和“不同的人有什么不一样”，了解本次合成场景出现的模式。再看“我们当初的猜测对不对”，确认模拟结果是否支持预先封存的假设。

最重要的是“这份报告能信多少”。报告中的机制解释可能只是合理推断，不是真实受访者已经说过的话。把其中标出的“需真人验证”内容整理成访谈提纲，才是下一步。

可以继续对 Agent 说：

```text
请根据 report.md 中需要真人验证的假设，生成一份 30 分钟访谈提纲。
```

## 常见问题

### 我只想设计问卷，不想跑完整流程

直接说明“只运行 WF1”。单步完成后不会自动生成 persona。

### 我已经有问卷

把问卷交给 Agent，要求先补齐研究问题、目标人群和每题测量构念，再从 WF2 继续。

### 为什么 Agent 要我补充目标人群经历

“20–30 岁”只能描述身份，不能解释为什么答案会不同。“近半年购买过家具的租房者”才提供了可以推导任务情境的经历。

### 为什么不能把合成结果写成真实调研

所有 persona 和回答都来自模型，没有经过真实人群抽样。它可以帮助检查问卷和准备假设，但不能替代真人证据。

### 验证脚本报错怎么办

把完整错误信息交给 Agent。`scripts/validate_run.py` 会指出缺失文件、字段或无法追溯的 id。不要让 Agent跳过错误或凭空补一份统计结果。
