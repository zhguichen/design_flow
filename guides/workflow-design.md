# design-flow Workflow 设计理念

这份文档解释完整流程为什么被拆成这些阶段，以及每个阶段试图防止什么错误。字段级合同以 `design_flow/workflows/` 和 `scripts/validate_run.py` 为准；这里讨论设计意图，不重复 schema。

## 总体原则

`design-flow` 的目标不是制造一批“看起来像真人”的数据，而是建立一条可追溯、可质疑、允许失败的合成预调研链路：

```text
问卷构念
→ 影响回答的输入变量
→ 可证伪的行为机制
→ 可观察的任务情境
→ 具体 persona
→ 隔离条件下的模拟作答
→ 确定性统计与解释
```

完整链路有 5 个 workflow 文件、7 个执行阶段：

| 阶段 | 核心问题 | 主要产出 |
|---|---|---|
| WF1 | 到底要量化什么 | `survey.json` |
| WF2 Phase A | 哪些输入差异会让回答不同 | `archetypes.json` |
| WF2 Phase B | 为什么这些差异值得纳入模拟 | `behavior_mechanisms.json` |
| WF2 Phase C | 哪些具体经历和限制影响任务判断 | `task_frictions.json`、`hypotheses.json` |
| WF3 | 如何把原型展开成有差异的具体人物 | `respondents.jsonl` |
| WF4 | 这些人物在不知道预测方向时会怎么回答 | `responses.jsonl` |
| WF5 | 模拟中出现了什么、如何解释、还要验证什么 | `stats.json`、`report.md` |

## WF1：先把测量对象说清楚

问卷不是问题清单，而是一组对已知构念的测量。WF1 强制每道题记录 `construct_measured`，因为后续人群分析不能只看题面猜测研究意图。

题序采用“筛选 → 行为 → 态度 → 开放”。行为先于态度，是为了给“满意吗”“愿意用吗”建立真实上下文；开放题放在最后，是为了避免在核心题之前消耗过多注意力。

这一阶段主要防止三类问题：研究问题其实适合访谈却硬做问卷；目标人群只有人口统计描述，没有相关经历；题目带有引导、双重提问或不可解释的量表。

WF1 不确认最终 `simulation_n`。在不知道原型数量时讨论生成多少人缺少依据，因此规模被推迟到完整 audience package 形成之后。

## WF2 Phase A：从回答差异反推人群

Phase A 不从年龄、性别、城市等属性做全排列，而是逐题追问：“什么经验、情境、心理、资源或能力会让这道题的回答不同？”

这种设计把 archetype 定义为“解释回答差异的最小分组”。人口统计只有在确实影响任务情境时才进入原型。

`scenario_weight` 是覆盖分配，不是人口比例。没有真实数据时使用等权覆盖，比让模型猜一个貌似真实的比例更诚实。`simulation_n` 同样只是计算预算；增加合成行数不会增加真人证据。

Phase A 的产出仍是候选。它必须接受 Phase B 的反向淘汰，而不是一旦生成就默认成立。

## WF2 Phase B：机制用于质疑，不用于证明

只有“忙碌型”“谨慎型”之类标签，仍然不能解释一个场景为什么可信。Phase B 要求把原型放进完整因果链：

```text
环境或事件
→ 社会处境
→ 资源与能力限制
→ 行为机制
→ 表层需求
→ 假设性潜在动机
```

机制库提供的是推理模板，不是目标人群存在的证据。因此每个机制必须标注证据等级，并同时写出替代解释和证伪条件。只依赖模型推理时，合理性最多是 `plausible`，不能写成 `supported`。

这一阶段的价值不是让报告显得更“有理论”，而是迫使 Agent 说明：为什么纳入这个场景、还有什么别的解释、未来看到什么真人证据时应该放弃它。

## WF2 Phase C：把性格标签还原成任务事实

“谨慎的人更担心安全”是人格直推，无法说明具体担心来自哪里。Phase C 把机制落到任务链和可观察 drivers，例如：

```text
与孩子同住
+ 经常负责做饭
+ 厨房空间狭小
+ 有热油和刀具风险
→ 与 risk_safety、space_environment 相关
```

这里不生成摩擦分数、top friction 或答案查找表。那些字段会把预期答案泄漏给后续模拟。Phase C 只提供能写进 persona 的经历、资源、能力和环境限制。

需要比较的预测被单独写进 `hypotheses.json` 并封存。WF3 和 WF4 都不能读取它；WF5 才第一次打开。这样模拟不支持预测时，可以保留失败，而不是重新生成人物直到“猜对”。

## WF3：具体化，而不是复制原型

WF3 将每个 archetype 展开成多个 persona。相同 archetype 共享关键处境，但任意两个个体至少在多个维度上不同，例如时间压力、预算、经验、渠道、认真程度或是否愿意承认不了解。

任务情境 drivers 会被改写成五层人物故事中的具体事实，不以独立分数表附在 persona 后面。完整机制定义也不复制到每一行；persona 只保存 `mechanism_ids` 和这个人特有的 `individual_expression`，运行 WF4 时再按 id 联结。

这样做同时解决两个问题：避免同类 persona 只是复制粘贴；避免大量重复字段占用上下文并在修改时发生版本漂移。

persona pool 生成后，用户可以选择全部模拟，或只指定 n 做 `stratified-pilot`。具体 respondent id 由 `scripts/select_respondents.py` 按 archetype 确定性抽取并记录 seed。用户不能手工挑 persona，否则容易在看到人设后选择更符合预期的人。pilot 必须覆盖所有 archetype，但仍标记为有限覆盖。

## WF4：把生成模型与预测方向隔离

WF4 采用“故事驱动”的方案。作答模型只看到问卷、persona 的可观察背景、个体化机制表现，以及按 id 联结的机制说明。它看不到 `hypotheses.json`、`task_frictions.json`、跨 persona 比较或任何答案方向。

这并不意味着回答是随机的。模型仍要体现有限理性、经验限制、不确定性和作答风格，并为关键题提供能追溯到人物经历的 reasoning。被禁止的是按“F2=5 所以选 5”执行规则表。

如果能在隔离 context 中运行，盲法标记为 `isolated`；如果只能在共享上下文中依靠程序性约束，标记为 `procedural` 并降低质量置信度。

## WF5：统计事实和解释判断分开

描述统计由 `scripts/analyze.py` 计算，避免模型手算均值、分布和交叉表。LLM 负责开放题主题编码、机制解释和设计建议。

分析报告必须分开三层：

1. 本次合成记录中实际出现的模式。
2. 行为机制与任务情境可能提供的解释。
3. 必须找真人才能确认的假设。

`hypotheses.json` 在这里首次打开。每条预测按实际结果标记为支持、不支持或无法判断。预测失败是有效结果，不允许为了报告好看而重跑模拟。

开放题分析采用跨受访者抽样原则。每条观察保留 source id，检查每个受访者都被覆盖，只出现一次的主题明确标为单源，避免前几个 persona 主导结论。

## 为什么只有三道确认门

确认点对应用户真正需要做的决策，而不是对应内部文件数量。

第一道门确认问卷，防止错误测量进入下游。第二道门一次查看完整 audience package 并确认生成预算，避免在 Phase A/B/C 之间重复询问。第三道门抽查 persona，确认模拟输入像目标用户，并选择 `full` 或 `stratified-pilot n`。

第三道门之后，WF4 和 WF5 连续运行。它们之间没有新的用户决策，增加暂停只会制造流程负担。pilot 报告必须显示 persona pool 数、实际模拟数和有限覆盖声明。

## 为什么机械校验独立成脚本

过去字段合同同时出现在根 Skill、workflow 和测试文档里，容易出现一处修改、其他地方仍保留旧规则。现在 `scripts/validate_run.py` 是结构、id、禁止字段、强类型和跨文件引用的单一机械事实来源。

workflow 只保留机器难以判断的内容，例如问卷是否中性、机制是否连贯、persona 是否自然。原则是“脚本检查事实，模型判断质量”。

## 最重要的边界

这条链路提高的是合成场景的结构质量和可审计性，不是研究效度。机制合理不代表人群真实存在，生成更多 persona 不代表样本更有代表性，模拟中的百分比也不能外推为现实市场比例。

最终设计决策仍应回到真人访谈、真实问卷或其他经验材料中验证。
