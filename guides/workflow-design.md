## 整体设计哲学

### 1. 这不是"让 AI 编一份问卷然后编一些答案"

Design Flow 的核心设计理念是：**模拟的质量取决于模拟者的质量。** 如果只是让 AI 随机生成几个年龄/职业标签然后直接填答案，AI 会退化为靠刻板印象瞎猜——"年轻人就选价格低的""中年人就选品质好的"。这不是模拟，这是概率猜测带点偏见。

因此，Design Flow 把 80% 的工作放在了**"制造一个能认真作答的人"上**，而只有 20% 在"让他填问卷"上。

### 2. 因果链驱动，不做属性穷举

传统 persona 方法容易退化为"给年龄、职业、收入贴一堆标签，然后凭直觉猜他会在乎什么"。Design Flow 改用一条可验证的因果链：

```
环境 / 事件
  → 社会处境（这个人正面临什么局面）
    → 资源 / 能力限制（他能做什么、不能做什么）
      → 行为机制（他为什么会这样想、这样做）
        → 表层需求（他说他要什么）
          → 假设性潜在动机（他可能真正需要什么）
```

每个 persona 的每个行为倾向都必须能沿着这条链往回追溯。走不完这条链的"人群类型"不会进入模拟——它只是可能存在的标签组合，但没有推导价值。

### 3. 防止自我实现的预言

如果你在生成虚拟受访者时，已经告诉他"你应该给满意度打 4 分"，那他打出来的 4 分就不能证明任何东西——你只是让自己的假设绕了一圈又回到了结果里。

Design Flow 通过三层隔离防止这个问题：
- **结果变量不进 persona 背景**：满意度/使用意愿/付费意愿永远只作为问卷题目出现，不作为人物设定
- **上游不生成答案方向**：不记录某类人应该打高分、选哪项或如何排序
- **不做规则查表**：WF4 不传摩擦分数或答案规则给模拟模型，只传“这个人是谁、经历过什么”

### 4. 确定性工具做事实，模型做判断

AI 模型擅长判断和推理，不擅长算术。Design Flow 把"谁说了算"分成两条线：
- **确定性的**交给 Python 脚本：统计计算（均值/中位数/分布）、JSON 结构校验、分层抽样
- **需要判断的**交给 LLM：因果链是否自洽、开放题的主题编码、报告中的洞察归纳

混在一起会出现 LLM 算错均值还自信满满地写进报告的灾难场景。

### 5. 渐进式披露

根 `SKILL.md` 约 113 行，只写"是什么、什么时候用、红线在哪"。具体的执行方法、合同定义、验收标准下沉到 5 个独立的 workflow 文件里，按需加载。

这样每个 Agent 进来只需要读完约 100 行的上下文，就能知道自己在干什么、不许干什么。需要执行某一步时，再加载对应的约 150-200 行 workflow 文件。

## Pipeline 全貌

```
WF1         WF2-A       WF2-B          WF2-C             WF3          WF4          WF5
问卷设计 → 人群反推 → 行为机制映射 → 任务摩擦映射 → Persona 生成 → 模拟作答 → 结果分析
  |           |           |              |                  |            |            |
survey.    archetypes. behavior_     task_frictions. respondents.  responses.  stats.json
json        json        mechanisms.  json              jsonl         jsonl       report.md
                        json
```

7 个执行阶段分布在 5 个 workflow 文件中（WF2 内含 A/B/C 三个 Phase）。

每步产出物都是结构化的 JSON/JSONL 文件，有明确的 schema 合约。下游步骤机械消费上游字段，不靠自然语言"理解"上游产出。

## WF1：问卷设计

### 它在解决什么

大多数人设计问卷时先想"我要问什么"，然后凭直觉列出十几道题。但问卷设计真正的起点是：**你想量化什么、谁来填、干什么用。**

### 设计关键

**三问前置。** 缺一个就停下来问用户，不猜。这不是不信任用户，而是避免跑偏浪费所有下游工作——如果问卷核心都没想清楚，后面 6 个阶段的产出都是废的。

**题序是有代价的。** 不是习惯问题。"筛选→行为→态度→开放"这个顺序是基于认知负荷设计的：开放题放前面会提前耗尽耐心；态度题没有行为上下文就变成空洞标签。

**construct_measured 是第一公民。** 这是整个 pipeline 最重要的字段。它不只是给题目分类，而是 WF2 反推人群的唯一入口——AI 从这里读"这道题测什么"，然后倒推"哪些人的答案会不一样"。填错了这个字段，整个人群分析就歪了。

**预设偏差检查表。** 不靠 AI"你觉得有没有偏差"——AI 说没有的时候可能只是没注意到。用具体反例检查：引导题、双重题、加载语言、回忆过载、术语。每条都有匹配的改法。

### 为什么 purpose 只能写 pre-research

这不是为了限制用户，而是为了**诚实面对 tool 的能力边界**。向用户坦诚说明：合成样本做不到真实假设验证或发表级研究。如果用户坚持要发表级结论，应该拒绝外推，而不是给 `purpose` 换一个漂亮的标签就假装能做到。

## WF2-A：人群反推

### 它在解决什么

传统做法是"我们假设有三类用户：年轻人、中年人、老年人"，然后往上贴标签。问题是人口统计标签解释不了"为什么同一年龄段的人答得不一样"。

### 设计关键

**从题目倒推差异来源。** 不凭空造人群类别。每道题问："哪些输入变量会让这道题的答案不同？"差异来源只有五类：经验、情境、心理、资源、能力。

**原型不是排列组合。** 候选原型必须满足三个条件才能保留：
1. 能解释同一道题为何有不同的答案
2. 经验、限制与资源之间自洽
3. 能在 Phase B 形成连贯、可证伪的机制链

通不过的合并或删除。不是所有可能性都值得跑一遍——只保留有解释力的。

**场景权重诚实标注来源。** 每个 `scenario_weight` 必须标注来源：用户真实数据？用户指定的？还是因为没依据所以等权覆盖？不让 `coverage-default` 伪装成 `real-data`。

## WF2-B：行为机制映射

### 它在解决什么

"这类人会更在乎价格"——这是直觉，不是分析。你需要解释**为什么**：他的处境、他经历过什么、什么机制在驱动这种倾向。

### 设计关键

**Domain 路由，不默认套技术接受模型。** 不是所有问卷都关于技术接受。问卷可能关于空间体验、产品购买、服务参与、习惯改变……先判断问卷属于哪个 domain，再从对应的机制池里选。用错机制（比如对空间体验问卷套 TAM）会导致推导脱节。

**每条因果链走完六个环节。** 不能跳过中间步骤。不能从"环境"直接跳到"表层需求"。走不完、不会影响回答、场景不自洽的原型要回退修改或删除。

**证据纪律。** 这是防止机制推导变成"我觉得是这样"的关键：
- 每条机制标注证据等级（user-evidence / cited-research / model-inference）
- `model-inference` 的 plausibility 最高只能是 `plausible`，不能升级为 `supported`
- 每个机制必须有竞争性的替代解释和可执行的证伪条件——说清楚"如果我错了，什么观察能证明我错了"

**Demand Authenticity Labels。** 表层需求不等于真实需求。区分 real_need / surface_need / contextual_need / compensatory_need / pseudo_need——这决定了后续设计建议的优先级。

## WF2-C：任务摩擦映射

### 它在解决什么

问"哪个环节最麻烦"时，不能靠"这个人比较谨慎/开放"来回答。性格只能影响如何应对摩擦，不能定义摩擦在哪。

### 设计关键

**从任务结构推导，不从人格猜测。** 链式推导：

```
任务场景 → 当前行为 → 环境限制 → 资源/能力 → 行为机制 → 摩擦维度
```

10 个稳定维度覆盖了大部分生活任务中的摩擦来源：time_effort / cognitive_load / risk_safety / money_loss / space_environment / identity_control / social_coordination / maintenance_aftercare / habit_disruption。

**drivers 必须是可观察的具体事实。** 不写"F2=5"（分数），不写"尺寸规划是最高痛点"（排名），不写"这个维度得分高所以答案应该是 X"（查表规则）。而是写"曾买回尺寸不合的家具、租房不可改造、空间规划能力中低"——这些是可写进 persona 故事的具体经历，WF4 看到后自己判断。

**不生成逐题预测。** Phase C 只记录哪些可观察经历、资源、能力和环境限制与题目有关，不预测哪类人一定更在意什么。答案方向留给 WF4 根据完整 persona 独立判断。

## WF3：Persona 生成

### 它在解决什么

"一个退休老人"不是 persona——标签太空了，没法用来模拟填答案。你需要一个具体的人：她住哪、每天早上干什么、买菜时走哪条路、为什么犹豫、上次退货是什么原因。

### 设计关键

**五层信息结构。** 每个人不只是一张人口统计标签，而是有完整的故事：
1. **基础身份**：年龄/职业/城市/收入（只有这层是不够的）
2. **社会处境**：正面临什么局面？时间压力、社交资源、家庭支持？
3. **心理倾向**：AI 信任度、风险偏好、从众还是独立判断？
4. **行为习惯**：平时找资料的方式、使用工具的习惯、购物路径？
5. **作答风格**：认真填还是应付？会不会选"不确定"？价格敏感度？

五层缺一不可。纯人口统计（只有第一层）的 persona 退化为标签，没法驱动模拟。

**任务摩擦 drivers 转写成事实。** `task_frictions.json` 中的 drivers 要写进 persona 的五层字段，变成具体的故事细节。但不保留分数、排名或答案规则——那些会泄漏预测方向。

**五个 Anti-pattern 自检。** 这不是走过场，是防止最常见的合成 persona 失败模式：
- Elastic（同类雷同）：同类 persona 变成复制品→模拟答案也雷同→统计上看不出差异
- Demographic-only（纯人口统计）：只有年龄职业→回答没有深度
- Ideal user（理想用户）：全是好用户→模拟不出负面反馈→回收的数据一片和谐但没用
- Committee（拼接怪）：一个 persona 塞了矛盾特征→回答不自洽
- Stale（过期）：无法追溯生成时间→不知道是哪个版本跑的

**同类个体强制差异。** 同一个 archetype 下的 persona 必须在至少 3 个维度上不同（比如专业方向、时间压力、经济条件、AI 信任度……）。否则模拟出来的答案会高度雷同，cross-tab 分析没有意义。

**置信度反映模拟质量，不是研究效度。** 三维评分（场景覆盖、数据质量、一致性），封顶 medium（单源 LLM 生成），并在 meta 中显式声明"不等于研究效度"。

## WF4：模拟作答

### 它在解决什么

让 AI 装成一个人填问卷很容易，但装得"像"这个人很难。如果给了太多信息（比如"你应该在 Q3 选 4 分"），就退化为查表；如果信息不够，就退化为随机生成。

### 设计关键

**Subagent 隔离。** 这是整个 pipeline 最关键的机制设计。每个 persona 的模拟填答在独立的 subagent 中运行——这个 subagent 只拿到：
- 该 persona 的五层背景信息
- mechanism_trace（心理机制描述）
- 所属 archetype 的 behavior_mechanisms（知道背后的因果关系，不只知道标签）
- survey.json

拿不到的是：
- 其他 persona 的信息（防止对比影响）
- task_frictions.json（防止摩擦分数泄漏答案方向）
- 任何"你应该选哪个"的规则或分值

**本质是即兴表演，不是剧本朗读。** "演员拿到角色档案后即兴表演——角色档案告诉你这个人是谁、经历过什么、在乎什么，但不会告诉你每句台词应该怎么念。"好的 reasoning 能追溯到 persona 的具体处境和经历，不是引用分数。

**有限理性而非无效作答。** 模拟的是"认真但有局限的人"——有偏见、会犹豫、有时不了解、有时保守——而不是"瞎填/反填/乱选"。无效作答不提供设计洞察。

**降级机制。** 如果当前环境不支持 subagent（比如某些 CLI 配置），降级为 procedural blinding，在 meta 中标记降级原因和置信度影响。不静默降级，不把共享上下文标为 isolated。

### 为什么不做查表

如果 WF4 拿到这样的信息："A1 类型在时间摩擦维度得分 5，所以 Q3 选 4"，那就不是模拟——是绕了一圈把自己的预期打印出来。真正的模拟是：告诉模型"这个人 deadline 很赶、导师催得紧、同学群发问卷没人回"，然后让模型自己判断他会不会在"效率 vs 规范"的排序题里把效率排第一。

## WF5：结果分析

### 它在解决什么

数据拿到了，但分析有两个常见的坑：统计算错了（LLM 算术不可靠），或者把合成场景里的模式写成真实人群结论。

### 设计关键

**先跑统计脚本，再做主题编码。** `scripts/analyze.py` 对 likert/rating/NPS 计算均值、中位数、分布；对分类题算频次；对排序题算平均排名；按 archetype 出 cross-tabs。LLM 再基于这些数字写报告，不自算任何统计量。

**不为形成漂亮结论而重写数据。** 先描述统计和回答原文中实际出现的模式，再判断机制是否能提供一种可能解释。解释与数据不一致时保留数据、降低解释置信度，不重生成答案。

**报告三层结构。** 读者要知道什么信到什么程度：
1. **模拟数据显示的模式**：这批合成数据里看到了什么
2. **机制推导出的解释**：用 WF2 推导的因果链来解释为什么
3. **需真人确认的问题**：哪些观察不能从合成数据外推，真实研究应优先检查什么

**Cross-Interview Sampling。** 强制交叉受访者：每条观察标 source id，验证每个受访者至少出现一次，单源模式标记。防止分析只用前几个 persona 的特征下结论。

## 关键架构决策

### 为什么用 JSON 合同而不是自然语言

上游产出散文，下游读散文找信息 → 信息丢失、格式不一致、字段缺失不可检测。

每步产出结构化 JSON，有明确的 schema。下游按字段名读取。`validate_run.py` 在每个阶段结束后做机械校验——缺字段、类型错误、ID 不一致全在进下一步之前抓到。LLM 偶尔会写错格式，机械校验做安全网。

### 为什么统计走脚本不走 LLM

"60 行数据的平均数和标准差"看起来简单，但 LLM 容易在细节上出错：忘了排除"不确定"选项、把 NPS 当 likert 算了、小数位数不一致……而且错了之后你很难发现（因为 LLM 写出来的文字很流畅，你不会怀疑每个数字的准确性交给确定性脚本，错误会是一致性错误而非随机错误，更容易抓到）。

### 为什么三道确认门放这而不是别处

不是所有步骤都值得停下来确认。三道门的放置逻辑：

| 门 | 为什么放这 |
|---|---|
| **问卷确认** | 问卷是整个 pipeline 的根数据。问卷歪了，六步全废。而且用户需要确认题目本身合理。 |
| **人群+规模确认** | 人群分析的产出直接决定了后续 persona 长什么样。此时 WF2 三个阶段全做完，信息最全，适合一次性展示。规模确认也只需要一次——不在 WF1 就确认是因为那时还不知道有多少 archetype。 |
| **Persona+模式确认** | 具体的人物形象用户需要抽查。模拟模式（full vs pilot）也要用户决定。 |

内部 Phase（WF2 的 A→B→C）自动推进不暂停，因为这是连续的推导链，中途停下来意义不大——只要验收通过就往前走。

### 为什么不允许多个独立注册的 Skill

根 Skill 是唯一入口，workflow 按需加载。这样：
- Harness 只暴露一个 Skill，用户不会被"选 WF1 还是 WF2"搞晕
- 按"我有个研究问题，想跑一遍"这种自然语言触发，比"先 load 01、再 load 02"的用户体验好得多
- 维护时只需要更新 workflow 文件，根 Skill 的触发词和路由表保持一致

## 红线的设计依据

每条红线背后都保护着一个具体的陷阱：

| 红线 | 防止什么 | 如果不防会怎样 |
|---|---|---|
| **所有输出标注合成样本/仅供预调研** | 合成样本伪装成真人证据 | 用户拿合成数据当真实数据做决策 |
| **结果变量不进 persona 背景** | 自证预言 | 设成"高满意"→填出"高满意"→"看，用户满意"→其实是自己骗自己 |
| **行为机制必须标证据等级+替代解释+证伪条件** | 把推测当实证 | "这类人存在"从一个合理推测悄悄变成既定事实 |
| **上游不生成逐题预测方向** | 自证预言 | persona 或机制文件已经告诉 WF4 应该选什么，模拟只是复述设定 |
| **task_frictions 不生成分数/排名/答案规则** | 查表式模拟 | persona 变成 friction 分数的排序结果，WF4 失去判断的独立性 |
| **用户只能选模式和 n，不能手挑 persona** | 确认偏误代入样本选择 | 用户只挑符合预期的人跑模拟，结果当然更接近预期 |
| **统计走脚本不走 LLM 自算** | 算术错误被流畅文字掩盖 | 报告文笔很好但根本数据是错的 |
| **不得为制造分群差异而重生成** | 数据粉饰 | “再跑一次，这次让 A 类人多打几个高分” |
| **场景权重不能解释为现实比例** | 合成数据冒充市场分布 | 报告中出现"占目标人群的 30%"这种无依据的数字 |
| **不模拟无效作答** | 噪声污染分析 | 瞎填的数据跟认真填的混在一起，分析不出有效信号 |

## 设计总结

Design Flow 本质上是一个**受控的幻觉系统**。它让 AI 产生对设计有用的"幻觉"（合成受访者、模拟填答），同时通过多层契约、验证和隔离机制控制幻觉的方向和质量。

它不追求"模拟像真人一样"，而是追求"模拟出来的偏差是可追溯、可检验、可证伪的"——你知道这个人的答案从哪来、为什么来、在什么条件下可能不同。这比"看起来像真人"更有工程价值。

如果你要扩展或修改这个 Skill，理解这些设计决策比记住具体字段名更重要。加新功能时先问：它会打破哪条因果链？它会绕过哪个隔离层？它会让哪条红线变模糊？
