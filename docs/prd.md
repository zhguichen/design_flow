# PRD: design-flow

## Summary

`design-flow` 是面向设计类学生 / 从业者的 Claude Code Skill，覆盖"设计问卷 → 反推人群 → 行为机制映射 → 任务摩擦映射 → 生成 persona → LLM 模拟填写 → 收集结果 → 分析"的完整链路。核心解决"缺少结构合理、场景覆盖充分的问卷预检输入"这个痛点——用有机制依据和任务摩擦依据的 AI 合成场景替代低质量学生式互填。

## Problem

设计类学生做问卷调研时普遍遇到：

- 找不到足够多的目标用户填问卷。
- 找到的人不是目标用户，只是同学、朋友、群聊互填。
- 填写者敷衍、乱填、反填，样本污染。
- 学生缺的不是问卷制作能力，而是可信、足量、结构合理的受访者样本。

结果：问卷数据质量差、分群混乱、结论不可信，毕业设计 / 课程作业的调研环节形同虚设。

## Target Users

- **设计类学生**（本科 / 研究生）做课程作业或毕业设计问卷调研
- **设计类从业者**做早期预调研 / 方向判断，无预算招真实样本
- **AI agent 维护者**想把"合成样本模拟"作为可复用能力接入自己的 context 基础设施

## Success Criteria

1. 用户给一个研究问题，skill 能产出一份结构良好的问卷 + 行为机制映射 + 任务摩擦映射 + 合成受访者 + 模拟数据集 + 分析报告，全链自洽。
2. 每类合成受访者都能追溯到“环境/事件 → 处境/限制 → 行为机制 → 任务摩擦 → 表层需求/假设性潜在动机”的因果输入链，而不是只靠人口统计拼 persona。
3. 痛点题、排序题、开放题的 persona 背景能追溯到 `task_frictions.json` 的可观察 drivers；WF4 只依据事实化的人物故事独立作答，不能只靠“性格”猜，也不能执行摩擦分数查表。
4. 合成受访者像真实目标人群：五层信息结构齐全、同类下个体有差异、作答体现有限理性与不确定性（非剧本式）。
5. 分析报告统计来自确定性脚本（非 LLM 自算），主题编码强制覆盖所有受访者（防前几个 persona 主导）。
6. 所有输出显式标注 合成样本 / 合成记录数 / 场景覆盖质量 / 仅供预调研；报告区分"模拟模式"、"机制与任务摩擦解释"与"需真人验证的假设"。
7. 根 skill 保持精简，专项内容下沉到 5 个按需加载的 workflow 文件；WF2 内含三个 Phase，全链共 7 个执行阶段；新用户一句话即可安装。
8. 预注册预测与模拟输入隔离：`hypotheses.json` 在模拟前封存，WF3 / WF4 不读取，WF5 才逐条对照；不支持假设时保留结果，不重生成。
9. `simulation_n` 仅表示计算预算，`scenario_weight` 仅表示场景分配且记录来源；无真实依据时等权覆盖，不输出总体比例或抽样精度结论。
10. 每个行为机制都有 `evidence_level`、`plausibility`、替代解释与证伪条件；理论卡片不作为目标场景证据，模型推断不升级为 `supported`。

## Non-Goals

- 替代真实用户研究或专业级样本采集
- 提供正式发表级统计推断（p 值、回归等）
- 问卷分发平台 / 收集真实填答
- 用合成样本当真人证据发论文或做决策

## Boundary

明确边界：**预调研工具，不替代真实用户研究。** 它替代的是低质量、随便发、没人认真填的学生式问卷调研，而不是专业研究中的真实样本采集。
